<objective>
Fix the site_settings table PK collision that prevents a second tenant from saving maintenance-mode settings. Deliver a migration that replaces the integer DEFAULT 1 primary key with a uuid primary key and adds a UNIQUE(tenant_id) constraint. Update the SiteSettings TypeScript type to match.

Purpose: The site_settings table was created with `id integer PRIMARY KEY DEFAULT 1`. Every new tenant INSERT tries to use id=1, colliding with the default tenant's row. The service layer (siteSettings.ts) already queries and saves by tenant_id, so the fix is entirely at the schema and type level.
Output: One migration file + one type file update. No application logic changes needed.
</objective>

Create the file supabase/migrations/20260216400000_fix_site_settings_pk.sql with the following exact content:

```sql
-- Migration: Fix site_settings PK from integer DEFAULT 1 to UUID
-- Resolves GAP-3: second tenant INSERT fails with PK collision (both tenants get id=1)
-- After this migration: each tenant gets a random UUID as PK; UNIQUE(tenant_id) enforces one row per tenant

BEGIN;

-- Step 1: Add a new UUID column that will become the primary key
ALTER TABLE public.site_settings
  ADD COLUMN id_new uuid DEFAULT gen_random_uuid() NOT NULL;

-- Step 2: Drop the existing PK constraint (no other table has an FK to site_settings.id)
ALTER TABLE public.site_settings
  DROP CONSTRAINT site_settings_pkey;

-- Step 3: Drop the old integer id column
ALTER TABLE public.site_settings
  DROP COLUMN id;

-- Step 4: Rename the new UUID column to id
ALTER TABLE public.site_settings
  RENAME COLUMN id_new TO id;

-- Step 5: Set the UUID column as the new primary key
ALTER TABLE public.site_settings
  ADD CONSTRAINT site_settings_pkey PRIMARY KEY (id);

-- Step 6: Add UNIQUE constraint on tenant_id (enforces one settings row per tenant)
ALTER TABLE public.site_settings
  ADD CONSTRAINT site_settings_tenant_id_unique UNIQUE (tenant_id);

COMMIT;
```

The migration is wrapped in a transaction. If any step fails (e.g., the PK constraint name differs), the entire migration rolls back safely.

Important notes:
- No other table in the schema has a foreign key referencing site_settings.id, so dropping the PK constraint is safe.
- Existing row data (is_customer_app_live, maintenance_title, etc.) is fully preserved — only the id column value changes.
- The existing default tenant row (tenant_id = '00000000-0000-0000-0000-000000000001') will receive a new random UUID as its id; no code queries by id, so this is harmless.
