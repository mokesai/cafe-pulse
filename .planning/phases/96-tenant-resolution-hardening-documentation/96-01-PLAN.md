Purpose: Defense-in-depth — ensure a deleted tenant (with deleted_at timestamp set) cannot have its cookie set and appear to "work" until RLS blocks data access. Both the primary soft-delete flag (deleted_at) and secondary exclusion flag (is_active) must be checked and set for robust isolation.

Output: Updated resolveTenantBySlug() query with deleted_at filter; updated deleteTenant() action to set is_active=false.

<task type="auto">
  <name>Add deleted_at filter to resolveTenantBySlug()</name>
  <files>src/lib/tenant/context.ts</files>
  <action>
Update the resolveTenantBySlug() function to add .is('deleted_at', null) filter after the existing .eq('is_active', true) filter.

Current query chain (line ~118-122):
```typescript
const { data, error } = await supabase
  .from('tenants')
  .select('*')
  .eq('slug', slug)
  .eq('is_active', true)
  .single()
```

Change to:
```typescript
const { data, error } = await supabase
  .from('tenants')
  .select('*')
  .eq('slug', slug)
  .eq('is_active', true)
  .is('deleted_at', null)
  .single()
```

Why this matters: resolveTenantBySlug() uses createServiceClient() which bypasses RLS. Without the deleted_at filter, a soft-deleted tenant with is_active=true would still resolve and have its x-tenant-id cookie set. The RLS policy on the tenants table filters deleted_at IS NULL, but that doesn't help here because service client bypasses RLS.

Pattern from existing code: This follows the same defense-in-depth pattern used in resendInvite() (actions.ts lines 175-177) which chains .eq('tenant_id', tenantId).is('deleted_at', null).
  </action>
  <verify>grep -n "\.is('deleted_at', null)" src/lib/tenant/context.ts</verify>
  <done>resolveTenantBySlug() query includes .is('deleted_at', null) after .eq('is_active', true) on the Supabase query chain</done>
</task>

<task type="auto">
  <name>Set is_active=false in deleteTenant()</name>
  <files>src/app/platform/tenants/actions.ts</files>
  <action>
Update the deleteTenant() Server Action to add is_active: false to the update payload.

Current update payload (line ~156-160):
```typescript
const { error: deleteError } = await supabase
  .from('tenants')
  .update({
    deleted_at: now,
    status: 'deleted',
  })
  .eq('id', tenantId);
```

Change to:
```typescript
const { error: deleteError } = await supabase
  .from('tenants')
  .update({
    deleted_at: now,
    status: 'deleted',
    is_active: false,
  })
  .eq('id', tenantId);
```

Why this matters: Setting is_active=false provides the second line of defense. If a bug or oversight causes the deleted_at filter to be missed in a query, the is_active filter (which already exists in resolveTenantBySlug()) would still block the deleted tenant from being resolved.

Defense-in-depth principle: Two independent filters (deleted_at IS NULL and is_active = true) mean a bug in either one doesn't create a resolution gap.
  </action>
  <verify>grep -A5 "status: 'deleted'" src/app/platform/tenants/actions.ts | grep "is_active: false"</verify>
  <done>deleteTenant() update payload includes is_active: false along with deleted_at and status='deleted'</done>
</task>
