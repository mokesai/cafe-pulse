  <action>
    Apply the standard pattern to all 19 invoice sub-routes and purchase order sub-routes.

    Standard pattern (same as invoices/route.ts list route):
    ```typescript
    import { getCurrentTenantId } from '@/lib/tenant/context'

    // In each handler, after auth check:
    const tenantId = await getCurrentTenantId()

    // Every .from() query gets:
    .eq('tenant_id', tenantId)

    // Every INSERT gets:
    { tenant_id: tenantId, ...otherFields }
    ```

    Key patterns for sub-routes that fetch by ID:
    - When fetching invoices/[id]: add .eq('tenant_id', tenantId) alongside .eq('id', id). This ensures that even if someone provides a valid invoice ID from another tenant, the query returns nothing.
    - When fetching invoice_items by [itemId]: add .eq('tenant_id', tenantId).
    - When fetching purchase_orders by [orderId]: add .eq('tenant_id', tenantId).
    - When fetching PO attachments, PO items, PO receipts: add .eq('tenant_id', tenantId).
    - For invoice upload routes: include tenant_id in the inserted invoice record.
    - For PO send routes: add tenant_id filter to the PO lookup.
    - For attachment routes: the attachment itself references a PO; add .eq('tenant_id', tenantId) to both the PO lookup and the attachment query.

    Read each file before editing. The edit pattern is mechanical: find every .from(...).select/update/insert/delete and add the tenant filter. For inserts, add tenant_id to the data payload.

    Note on invoice items cross-reference: invoice_items may also have an invoice_id FK. When you add .eq('tenant_id', tenantId), that is sufficient — you do NOT need to also verify the invoice belongs to the tenant separately (the tenant_id column on invoice_items is the source of truth).

    Do not change: storage bucket queries for file uploads (storage is scoped by bucket policy from Phase 30), supabase.auth.* calls, Square API calls.
