  <action>
    Create `tests/e2e/isolation/menu-isolation.spec.ts`:
    - Test that tenant-a.localhost:3000/menu shows only Tenant A items
    - Test that tenant-b.localhost:3000/menu shows only Tenant B items
    - Use test.describe.configure({ mode: 'parallel' }) for concurrent execution
    - Follow Pattern 1 from 70-RESEARCH.md (getTenantForWorker helper)

    Create `tests/e2e/isolation/checkout-flow.spec.ts`:
    - Concurrent checkout test: Tenant A and Tenant B place orders simultaneously
    - Each worker: navigate to menu → add item to cart → open cart modal → verify cart count
    - Verify cart data is isolated (Tenant A cart !== Tenant B cart)
    - Do NOT complete payment (Phase 70 is isolation testing, not payment testing)
    - Follow Example 1 from 70-RESEARCH.md for concurrent test structure

    Create `tests/e2e/isolation/admin-isolation.spec.ts`:
    - Test that Tenant A admin user cannot access http://tenant-b.localhost:3000/admin
    - Test that Tenant B admin user cannot access http://tenant-a.localhost:3000/admin
    - Verify 403 or redirect to login when accessing wrong tenant's admin
    - Use page.getByText or page.locator to verify error messages

    All tests must:
    - Use subdomain routing (tenant-a.localhost:3000, tenant-b.localhost:3000)
    - Include expect() assertions for visible elements
    - Handle async navigation with await page.goto()
    - Include descriptive test names explaining the isolation scenario

    Note: Tests will FAIL initially if tenants don't exist in database. Document this in test comments as a prerequisite (run seed script or create tenants via platform admin).
  </action>
  <verify>

    - Check each test file imports `{ test, expect } from '@playwright/test'`
    - Verify subdomain patterns in test URLs (tenant-a.localhost:3000, tenant-b.localhost:3000)
  </verify>
