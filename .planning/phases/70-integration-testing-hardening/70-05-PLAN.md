<objective>
Resolve the WARNING gap in the 70-02 security audit: the __siteStatusCacheEdge cache uses a singleton pattern (one cached value shared across all tenants) instead of a tenant-scoped Map. This plan makes the decision (per-tenant) and implements it.

Purpose: Multi-tenant SaaS requires each tenant to independently control their own maintenance mode. A singleton cache means if Little Cafe goes into maintenance, any future second tenant would also appear in maintenance — wrong behavior. The site_settings table already has tenant_id from Phase 20; the cache just needs to be per-tenant.

Output: siteSettings.edge.ts refactored to Map<string, CacheEntry>; siteSettings.ts queries filtered by tenant_id; middleware callers updated to pass tenantId.
</objective>

<task type="checkpoint:decision" gate="blocking">
  <decision>Should maintenance mode (site status) be per-tenant or global?</decision>
  <context>
    The site_settings table has a tenant_id column (added in Phase 20 migration). The current siteSettings.edge.ts cache uses a singleton (one value for all requests). The current siteSettings.ts queries filter by id=1 (the original single-tenant row), not by tenant_id.

    Option A (per-tenant): Each tenant independently controls their maintenance mode. Tenant A can take their storefront offline without affecting Tenant B. Requires: refactor cache to Map keyed by tenantId, update siteSettings.ts to filter by tenant_id, update middleware to pass tenantId to getCachedSiteStatus.

    Option B (global/platform): One maintenance mode setting that affects all tenants simultaneously. Useful if the entire platform goes down (infrastructure maintenance). Simpler — keep singleton cache, document it. Platform admin controls the single setting.

    Context: This is the Little Cafe at Kaiser Permanente. The roadmap shows we're building toward multi-tenant SaaS. The site_settings table already has tenant_id. The natural SaaS behavior is per-tenant maintenance mode.

    Recommendation: Option A (per-tenant). This is a multi-tenant SaaS platform. Each cafe should be able to independently manage their storefront availability.
  </context>
  <options>
    <option id="option-a">
      <name>Per-tenant maintenance mode</name>
      <pros>Each tenant independently controls availability; correct SaaS model; site_settings.tenant_id column is used properly</pros>
      <cons>Slightly more complex cache; middleware must pass tenantId to getCachedSiteStatus</cons>
    </option>
    <option id="option-b">
      <name>Global/platform maintenance mode</name>
      <pros>Simpler; useful for infrastructure-level outages; no cache refactor needed</pros>
      <cons>Wrong for SaaS — one tenant's maintenance affects all tenants; contradicts multi-tenant design; wastes the tenant_id column</cons>
    </option>
  </options>
  <resume-signal>Select: option-a or option-b</resume-signal>
</task>
