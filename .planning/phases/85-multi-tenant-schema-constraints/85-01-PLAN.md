<objective>
Replace the three single-column UNIQUE constraints in the KDS domain with composite (tenant_id, field) constraints so two tenants can store KDS data with the same keys/filenames/variation IDs without conflict.

Purpose: Removes the KDS portion of GAP-2, which blocks multi-tenant KDS operation. Currently kds_settings.key UNIQUE means a second tenant cannot store any setting that the first tenant also has.
Output: Migration file `20260217000000_composite_kds_unique_constraints.sql` applied to Supabase.
</objective>

**kds_settings** — inline UNIQUE on `key` column, so use ALTER TABLE DROP CONSTRAINT:
```sql
ALTER TABLE public.kds_settings DROP CONSTRAINT kds_settings_key_key;
ALTER TABLE public.kds_settings
  ADD CONSTRAINT kds_settings_tenant_key_unique UNIQUE (tenant_id, key);
```

**kds_images** — named UNIQUE constraint, so use ALTER TABLE DROP CONSTRAINT:
```sql
ALTER TABLE public.kds_images DROP CONSTRAINT kds_images_filename_unique;
ALTER TABLE public.kds_images
  ADD CONSTRAINT kds_images_tenant_filename_unique UNIQUE (tenant_id, filename);
```

**kds_menu_items** — this is a partial index (NOT a constraint), so use DROP INDEX + CREATE UNIQUE INDEX:
```sql
DROP INDEX IF EXISTS idx_kds_menu_items_square_variation_id_unique;
CREATE UNIQUE INDEX IF NOT EXISTS idx_kds_menu_items_tenant_variation_unique
  ON public.kds_menu_items (tenant_id, square_variation_id)
  WHERE square_variation_id IS NOT NULL;
```
