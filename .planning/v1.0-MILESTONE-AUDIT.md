## Critical Gaps

### GAP-1: process-payment inserts orders without tenant_id (SC1 blocker)

**File:** `src/app/api/square/process-payment/route.ts:178–193`

The route resolves `tenantId` at line 39 and uses it for Square config (correct), but the `orders` INSERT payload does not include `tenant_id`. The column has `DEFAULT '00000000-0000-0000-0000-000000000001'`, so:

- Default tenant checkouts succeed — DEFAULT fills in the correct tenant ID
- Second tenant checkouts silently mislabel orders — orders from tenant B are stored with the default tenant's UUID

The route also uses `createClient()` (not `createTenantClient()`), so the Supabase client doesn't carry the `x-tenant-id` header either, meaning RLS policies enforcing `app.tenant_id` are not applied to this client.

**Fix:** Add `tenant_id: tenantId` to the orders insert payload at line 182. Also add `tenant_id` to the `order_items` map at line 208. Consider using `createTenantClient(tenantId)` to enable RLS scoping.

---

### GAP-2: 13 single-column UNIQUE constraints block simultaneous multi-tenant data (SC1 blocker)

STATE.md correctly flags this as "15+ tables have single-column UNIQUE constraints that will block multi-tenant data (deferred to Phase 60+)." These constraints were never updated to composite `(tenant_id, field)` form in Phase 20 or any subsequent phase.

**Confirmed blockers** (will prevent second tenant from inserting operational data):

| Table | Constraint | Blocker Scenario |
|-------|-----------|-----------------|
| `kds_settings` | `key text UNIQUE` | Both tenants need `image_rotation_interval` — always conflicts |
| `inventory_items` | `name text NOT NULL UNIQUE` | Tenant B can't have "Oat Milk" if Tenant A does |
| `suppliers` | `name text NOT NULL UNIQUE` | Tenant B can't have "Sysco" if Tenant A does |
| `measurement_units` | `name UNIQUE`, `symbol UNIQUE` | Both tenants need "lbs", "oz", etc. |
| `cogs_products` | `square_item_id UNIQUE` | Cross-tenant if same Square catalog synced |
| `cogs_product_variations` | `square_variation_id UNIQUE` (×2) | Same |
| `cogs_modifier_lists` | `square_modifier_list_id UNIQUE` | Same |
| `cogs_modifier_options` | `square_modifier_id UNIQUE` | Same |
| `kds_images` | `filename UNIQUE` | Two tenants can't share a KDS image filename |
| `kds_menu_items` | `square_variation_id UNIQUE` | Same as cogs |
| `cogs_products` | `product_code UNIQUE` (partial index) | Tenant B can't reuse codes like LATTE_12OZ |
| `purchase_orders` | `order_number UNIQUE` | Rare but possible collision |
| `webhook_events` | `event_id UNIQUE` | Square event IDs are globally unique per-event, so this is unlikely to conflict in practice |

**Most severe:** `kds_settings.key` — this **immediately** breaks KDS configuration for any second tenant with zero workaround. Every KDS setting key conflicts.

**Fix:** For each table, DROP the single-column UNIQUE constraint and CREATE a composite `UNIQUE(tenant_id, field)` constraint. For foreign-key-based constraints (like `(supplier_id, invoice_number)` where supplier_id already scopes to tenant), no change needed.

### GAP-3: site_settings id DEFAULT 1 prevents second tenant settings (SC1 blocker)

**Migration:** `supabase/migrations/20250829090000_create_site_settings_table.sql:3`

```sql
id integer PRIMARY KEY DEFAULT 1
```

The `site_settings` table uses an integer primary key with `DEFAULT 1`. Phase 70-05 correctly updated `siteSettings.ts` to query by `.eq('tenant_id', tenantId)` instead of `.eq('id', 1)`, which resolved the read path.

However, the `saveSiteSettings()` function does an INSERT with no explicit `id` value. PostgreSQL will use `DEFAULT 1`, causing a primary key violation for any second tenant's first INSERT.

**Evidence:** `src/lib/services/siteSettings.ts:111–115` — INSERT payload includes `tenant_id` but no `id`.

**Fix:** Change the `site_settings` PK to `uuid DEFAULT gen_random_uuid()` PRIMARY KEY, or add a `UNIQUE(tenant_id)` constraint and change insert logic to upsert on `tenant_id`.

### GAP-4: createTenant() does not create admin user (SC4 blocker)

**File:** `src/app/platform/tenants/actions.ts:104–105`

```typescript
// 4. TODO: Create admin user account
```

The onboarding wizard collects `admin_email`, validates it, but discards it. No Supabase auth user is created, no invite email is sent, no `tenant_memberships` record is inserted for the new tenant.

**Impact:** A newly onboarded tenant has:
- ✓ A tenant record in the `tenants` table
- ✓ Square credentials in Vault (after OAuth)
- ✗ No admin user who can log in to the admin dashboard
- ✗ No `tenant_memberships` row for the new tenant
- ✗ `requireAdmin()` always returns `?error=no-access` for the new tenant subdomain

**Fix:** After `createTenant()` succeeds, either:
1. Call Supabase Admin API `auth.admin.inviteUserByEmail()` to send an invite, then create a `tenant_memberships` row
2. Or create the user via Supabase Admin API and insert into `tenant_memberships` with `role = 'owner'`

## Security Gaps

### SEC-1: OAuth callback has no authentication check

**File:** `src/app/api/platform/square-oauth/callback/route.ts`

The authorize route correctly calls `requirePlatformAdmin()` before redirecting to Square. But Square's callback redirects back to the callback route directly, which has no authentication check. Any unauthenticated caller who constructs a valid `state` string and an auth code can invoke this endpoint to overwrite a tenant's Vault credentials.

This is partially mitigated by the unimplemented CSRF state token verification (TODO at line 42 of callback route) — if state verification were implemented with server-side storage, the attack surface would be reduced.

**Severity:** High — credential overwrite possible without authentication.

**Fix:** Add `await requirePlatformAdmin()` as the first call in the callback route handler.

### SEC-2: Platform Server Actions lack internal auth

**File:** `src/app/platform/tenants/actions.ts`

All five Server Actions (`createTenant`, `updateTenant`, `changeStatus`, `deleteTenant`, `restoreTenant`) call `createServiceClient()` directly but none call `requirePlatformAdmin()`. They rely on the layout's `requirePlatformAdmin()` running first in the UI flow.

Next.js Server Actions can be called directly via POST to their action endpoint by anyone who knows the action ID. Without auth inside the action, a crafted POST can create, delete, or change tenant status without any authentication.

**Severity:** Medium — not an immediate exploit through normal browsing, but exploitable via direct POST.

**Fix:** Add `await requirePlatformAdmin()` as the first call in each Server Action.

## What Works

The multi-tenant architecture is fundamentally sound. The wiring between phases is correct:

- **Tenant isolation at DB level** — 202 RLS policies, SECURITY DEFINER functions, storage policies all correctly use `app.tenant_id` session variable
- **Tenant context propagation** — middleware → cookie → client is properly connected across all phases
- **Square credentials** — 23 API routes use per-tenant credentials from Vault
- **Admin auth** — requireAdmin() correctly checks tenant_memberships (not profiles.role)
- **Email branding** — React Email templates use tenant identity
- **Platform control plane** — Full CRUD, state machine, MFA enforcement, Square OAuth
- **Caching** — All three cache layers (tenant, Square, site status) are tenant-scoped
- **Security audit** — 79/82 service-role queries audited and passing (3 are documented false-positives)

The 4 critical gaps above are surgical fixes, not architectural redesigns. Each can be addressed with focused migrations and small code changes.
